📖 L10: Industrial Graphics with Objects

AVEVA™ Operations Management Interface 2023 — Module 3, Section 3

Pages 183–186 (Section 3-45 to 3-48)

Introduction

When working with automation objects in AVEVA OMI, you need a way to associate visual symbols with the objects that represent real-world equipment. There are two distinct approaches to accomplish this: you can create symbols directly within an automation object, or you can link to external symbols that already exist on the Graphics tab. Understanding when and how to use each method is fundamental to building effective industrial graphics.

When you add a symbol or link an external symbol in a template object, that symbol is automatically propagated to all derived objects. The symbols inherited from a template object cannot be removed by downstream users — they are part of the template's definition. This inheritance model ensures consistency across your entire object hierarchy.

Linked Symbols vs. Added Symbols

Reference: Section 3-45 (Page 183)

Key Concept — Two Ways to Associate Symbols with Objects
You can attach visual symbols to automation objects using either linked symbols (creating a reference to an existing symbol on the Graphics tab) or added symbols (creating a new symbol natively hosted by the object). Each approach has different implications for how changes propagate, how symbols are managed, and what happens when objects are deleted.
AspectLinked SymbolsAdded Symbols
How it works Creates an association between the automation object and a symbol that lives on the Graphics tab Creates a new symbol that is natively hosted inside the automation object
Sharing The same Graphics tab symbol can be linked to multiple automation objects, even across different derivation hierarchies Each object hosts its own copy of the symbol; no sharing across objects
Editing Edit the symbol from the Graphics tab — changes apply to all linked objects simultaneously Edit the symbol within the object (you can embed a Graphics tab symbol and customize it)
Change propagation Changes to the Graphics tab symbol propagate to all linked automation objects Changes to a template's symbol propagate to child objects wherever referenced
Deletion behavior Deleting the automation object does not delete the symbol — it still exists on the Graphics tab Deleting the automation object also deletes the symbols, since they only exist within the object
Customization Custom properties in a linked symbol cannot be customized directly Fully customizable — edit properties as needed for the specific object
Tip — Which approach to choose? Use linked symbols when you want a single symbol definition shared across many objects (e.g., a standard valve symbol used by every valve in your plant). Use added symbols when you need per-object customization that goes beyond what a shared symbol can provide.

Additional notes about linked symbols:

  • Symbols on the Graphics tab can be linked to automation objects
  • Linked symbols in automation objects can themselves be embedded into other symbols
  • Changes made to a linked or embedded symbol propagate to all automation objects to which the symbol is linked and to all graphics in which the symbol is embedded
  • Relative references used in a linked symbol will be resolved through the object to which it is linked
Limitations of Linked Symbols: Custom properties in a linked symbol cannot be customized directly. Additionally, only symbols from the Graphics tab can be linked to an object — you cannot link symbols from other automation objects' Content panes.

Managing Symbols from the Object Editor

Reference: Section 3-46 (Page 184)

When you edit an automation object in the Object Editor, you manage its symbols through the Content pane on the Attributes tab. This pane provides two primary actions for adding visual content to your object.

The Content pane in the Object Editor — the "Add" button (left "+" icon) creates an empty symbol attached to the object, while the "Link Content" button (right "+" icon) opens the Galaxy Browser to link an existing symbol from the Graphics tab. (Section 3-46)
ButtonWhat It DoesResult
Add Creates an empty symbol attached to the automation object The new symbol appears in the Content pane and can be edited in the Graphic Editor
Link Content Opens the Galaxy Browser to browse and select a symbol from the Graphics tab A reference to the selected symbol appears in the Content pane; the symbol itself stays on the Graphics tab

From within the Object Editor, you can also rename symbols, add or edit descriptions, duplicate symbols, and delete symbols. To edit a symbol, double-click it in the Content pane or click the Edit button to open the Graphic Editor. For linked symbols, you can alternatively open and edit them directly from the Graphics tab.

Relative References

Reference: Section 3-47 (Page 185)

Relative references are a powerful mechanism for navigating the object hierarchy. Instead of using absolute paths to reference attributes, you use short reference strings that automatically resolve based on where the symbol is being used. This makes symbols portable and reusable across different objects in your hierarchy.

Key Concept — Valid Relative References
The following reference strings all resolve to the current object when used in a symbol:

Me, MyArea, MyContainer, MyEngine, MyHost, MyPlatform

These references can be used to configure symbols, and when the symbols are placed in objects at runtime, those relative references are resolved by the objects automatically.

Relative references are especially important when working with containment relationships. In a containment hierarchy, objects are organized with parent containers holding child objects. Symbols in each object can be embedded in other objects within the same containment using relative references, and the references within the embedded symbols resolve automatically without further configuration.

For example, if a contained object includes a symbol configured with relative references such as Me.PV, that symbol can be embedded in a symbol of the container object. At the template level, if a container template needs to include symbols from its contained templates, you add a symbol to the container template and then embed the contained templates' symbols through relative referencing.

Quiz

Question 1

What is the key difference between a linked symbol and an added symbol in terms of deletion behavior?

Click to reveal answer

When you delete an automation object that has linked symbols, the symbols are not deleted — they still exist on the Graphics tab. When you delete an object with added symbols, those symbols are also deleted because they only exist within that object.

Question 2

What are six valid relative reference strings that resolve to the current object?

Click to reveal answer

The six valid relative references are: Me, MyArea, MyContainer, MyEngine, MyHost, and MyPlatform.

Question 3

What happens to symbols inherited from a template object when you try to remove them from a derived object?

Click to reveal answer

Symbols inherited from a template object cannot be removed from derived objects. They are part of the template's definition and are automatically propagated to all derived objects.

Question 4

Why can't you customize custom properties in a linked symbol directly?

Click to reveal answer

A linked symbol is a reference to a shared symbol on the Graphics tab. Since the same symbol is shared across multiple automation objects, customizing its properties for one object would affect all other objects linked to that symbol. This is a fundamental limitation of the linked symbol approach — for per-object customization, you must use added symbols instead.

Key Takeaways

  1. Linked symbols reference a shared symbol on the Graphics tab; added symbols are hosted natively within the automation object.
  2. Linked symbols allow one symbol to serve many objects; added symbols allow per-object customization.
  3. Deleting an object with linked symbols preserves the symbol; deleting an object with added symbols destroys the symbol.
  4. Relative references (Me, MyArea, etc.) allow symbols to be portable across objects in a hierarchy.
  5. Symbols inherited from a template object cannot be removed from derived objects.
