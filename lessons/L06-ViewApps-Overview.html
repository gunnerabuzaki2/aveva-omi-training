📖 L06: ViewApps Overview

Based on Module 2 — Getting Started: Section 4 — ViewApps (pages 2-53 through 2-64)

The ViewApp is the central object in AVEVA Operations Management Interface. It brings together screen profiles, layouts, navigation, and content into a cohesive application that operators use to monitor and control their processes. This lesson covers how ViewApps are created, configured, navigated, previewed, deployed, and launched.

1. The ViewApp Object

Source: page 2-53

$ViewApp is provided in the Galaxy as a base system object to create Operations Management Interface applications. A derived ViewApp template can be configured in the ViewApp Editor for creating a view application as needed. After a derived ViewApp object is configured, an instance can be derived and deployed to run the view application.

Key Concept: The ViewApp creation process follows the standard template-instance pattern: derive a template from $ViewApp, configure it in the ViewApp Editor, then derive an instance from the template for deployment.

2. ViewApp Initialization

Source: pages 2-53 to 2-54

The first time you open a derived ViewApp template, the ViewApp Initialization wizard appears. This wizard guides you through selecting a screen profile and associating layouts with screens.

Wizard Step 1: Select Screen Profile

On the first wizard page, you select a screen profile from the available profiles in the Galaxy. The screen profile determines how many monitors the ViewApp will use and their physical arrangement.

The ViewApp Initialization wizard Step 1: select a screen profile. The available profiles are listed with thumbnail previews.

Wizard Step 2: Associate Layout

On the second wizard page, you select a screen and assign a layout to it. You can assign layouts to multiple screens, but at least one screen must have a layout assigned. Once you click Finish, the ViewApp Editor opens.

Step 2: select a screen and associate a layout with it. The left panel shows the screen selection; the right panel shows available layouts.
Tip: The initialization wizard only appears the first time a derived ViewApp template is opened. However, you can edit or change screen profiles and layouts later from the ViewApp Editor.

3. The ViewApp Editor

Source: pages 2-54 to 2-55

The ViewApp Editor is the primary interface for building and configuring a ViewApp. You can perform the following operations:

  • Configure and customize the navigation model
  • Edit or change the screen profile associated with the ViewApp
  • Edit, change, or remove a layout associated with the ViewApp
  • Edit or add content to layout panes
  • Test the ViewApp in preview mode without deploying or redeploying

Editor Components

The ViewApp Editor interface: (1) Navigation List showing the hierarchy, (2) Actions List showing pane content, (3) screen profile and layout preview, (4) tabbed area with Properties/Toolbox/Assets, (5) Preview commands.
#ComponentPurpose
1Navigation ListShows the hierarchy of items in the navigation model. The default is based on the plant model but can be customized.
2Actions ListShows content assigned to the panes of the layout applied to the selected screen.
3Screen Profile / Layout PreviewShows the screen profile and layout associated with the ViewApp, with zoom and pan controls.
4Tabbed AreaProperties tab (navigation item and content properties), Toolbox tab (symbols and controls from Graphics), Assets tab (Model view of Galaxy).
5Preview CommandsAllow you to view the ViewApp in a preview session and enable/disable automatic updates.

4. ViewApp Navigation

Source: pages 2-55 to 2-56

The navigation mechanism within a ViewApp controls the presentation of content at runtime. For example, a tree control shows the hierarchical organization of the built-in navigation. Selecting an item from the tree displays corresponding content in one or more panes of the ViewApp layout.

Three Essential Components of Navigation

ComponentDescription
Navigation ModelDefines a hierarchy of navigation items, references to visual content, and actions for navigation-aware controls. Drives the presentation of content from layout to screen.
Navigation ItemA unique element in the navigation model containing visual representation info and triggers for navigation actions after being selected at runtime.
Navigation ActionsRuntime behavior of navigation-aware controls to show information present in the navigation model.

Built-In Navigation

By default, the built-in ViewApp navigation model reflects the plant model for the Galaxy. In the navigation list, the root is named Home with Assets shown as a top-level item. The navigation is fully customizable — you can add and remove custom navigation items, and the root item can be renamed (though not deleted).

The ViewApp navigation model mirrors the System Platform Model view. Alias names assigned to objects in the Model view appear in the navigation model of a running ViewApp.
Key Concept: The navigation model is automatically generated from the Galaxy's plant model. Objects organized under areas in the Model view appear as hierarchical navigation items in the ViewApp. You can customize this hierarchy, but the default structure provides immediate, meaningful navigation for operators.

5. Auto-Fill

Source: pages 2-57 to 2-59

Auto-Fill is a navigation behavior that automatically places content in panes without having to configure each view of a running ViewApp. It requires two conditions:

  1. Layout panes must have the Use For Auto-Fill property enabled
  2. Navigation items must have Auto-Fill Content configured to something other than "None"

Auto-Fill Content Options

OptionBehavior
Current onlyContent from the current navigation item is automatically placed in available panes. If no layout is assigned, the Home item's layouts are used.
Current, then upSearches the current item first, then moves up the hierarchy to find content for empty screens. Content is placed using the best available pane algorithm.
Current, up, then downSimilar to "Current, then up," but also checks the first child at each level going down until reaching a leaf node.
NoneNo content is automatically shown. Content must be manually assigned to specific panes.

Auto-Fill Matching Behavior

Content display at runtime is based on three factors: the content types of content found from navigation, the available panes configured for those specific content types, and the Auto-Fill settings. When a content item's type matches a pane's configured type, the content fills that pane automatically.

Auto-Fill matching: content items with types (Overview, Detail, Trend) are matched to panes configured with the same content types. Panes are filled in alphabetical order when multiple panes share the same type.
When multiple panes have the same content type but only one symbol matches, the alphabetically first pane (Pane01) receives the content. Tabbed or Multiple presentation panes can accept multiple content items.
Auto-Fill with "Current, then up": when the current item's Trend symbol fills Pane03 first, the parent item's Overview and Detail symbols fill Pane01 and Pane02 respectively.

6. Preview Mode

Source: pages 2-59 to 2-60

You can use the preview mode from the ViewApp Editor to test your ViewApp without deploying or redeploying it to runtime. The preview session launches immediately and runs in a separate window to show the current content and ViewApp configuration.

Tip: Preview mode is intended for testing purposes only — it is not intended for production use.

Starting a Preview Session

Click the Preview button in the top-right corner of the ViewApp Editor to view the ViewApp in a native ViewApp window. By default, switching to runtime preview saves all current changes.

Updating a Preview Session

If a preview session is active and unsaved changes exist, the Preview button transforms to an Update button. Clicking Update saves changes and refreshes the preview session.

Automatic Update Mode

Preview mode supports automatic update, enabled by default. Toggle it via the refresh icon to the left of the Preview button:

  • Enabled: Changes saved to the ViewApp are automatically reflected in the preview session.
  • Disabled: Changes must be manually pushed by clicking Update.

Closing a Preview Session

Click the Close button (× icon) in the upper-right corner of the preview display. Note that closing the ViewApp Editor also closes the preview session.

7. Deriving and Deploying a ViewApp Instance

Source: pages 2-60 to 2-61

Once the derived ViewApp template is configured, you create a ViewApp instance from it. Right-click the derived ViewApp template and select New → Instance. In the Deployment view, the ViewApp instance must be hosted by a ViewEngine object before it can be deployed.

Important Rules:
  • You cannot derive an instance directly from the $ViewApp base system object — it must come from a derived template.
  • In Deployment view, only ViewEngine objects can host ViewApp objects.
  • The host ViewEngine object must be deployed before ViewApp instances can be deployed.

Deployment Process

ViewApp instances are deployed from the System Platform IDE like other objects. Right-click the ViewApp instance in Deployment view and select Deploy. Application files are asynchronously transferred to the target node — an orange icon appears during transfer, changing once complete.

Multiple instances of the same ViewApp can run on one node. If changes are made to the derived template while an instance is deployed, the icon indicates "deployed, but pending configuration changes."

Deployment status icons: the orange circular-arrow icon indicates files are being transferred; the blue window icon indicates the transfer is complete and the ViewApp is ready to launch.
When a ViewApp is running and its template has changed, this dialog allows you to update the running ViewApp immediately.

8. Launching a ViewApp

Source: pages 2-62 to 2-63

Deployed ViewApps can be launched from two application managers:

AVEVA OMI Application Manager

This manager is specifically for OMI ViewApps. ViewApps are listed by instance name, file path, size, and deployment date/time. Multiple ViewApps can be deployed and running on the same node.

The AVEVA OMI Application Manager: ViewApps are listed with instance names, file paths, and deployment details. Click LAUNCH to start the ViewApp.

AVEVA Application Manager

This general-purpose manager can launch both OMI ViewApps (on the OMI tab) and InTouch applications. Click the Details button to view file paths, deployment dates, and other information.

The AVEVA Application Manager with the OMI tab selected, showing deployed ViewApps with thumbnail previews. The OMI Viewer button launches the selected app.
Key Concept: The AVEVA OMI Application Manager is for OMI ViewApps only. The AVEVA Application Manager can launch both OMI and InTouch applications. Use the appropriate manager for your application type.

9. TitleBarApp

Source: page 2-64

The TitleBarApp is an OMI App that shows title bar information in a running ViewApp. It is typically placed in a pane at the top of the ViewApp window and provides functional components like Home, Login, Language, and Keyboard buttons.

The TitleBarApp in the Toolbox tab. Select it and drag TitleBarControl to a pane to add a title bar to your ViewApp.

TitleBarApp Configuration Categories

CategoryDescription
AppearanceVisual characteristics: foreground/background color, font, secondary row settings.
Content and Display AreaContent displayed and locations (e.g., Home button position on the left side of the primary row).
Window ButtonsEnable minimize, maximize, and close buttons. Close buttons include options to close the ViewApp or close the layout window.

Quiz

Test Your Knowledge

1. What is the name of the wizard that appears the first time you open a derived ViewApp template?

The ViewApp Initialization wizard. It guides you through selecting a screen profile and associating layouts with screens.

2. What are the five main components of the ViewApp Editor?

(1) Navigation List, (2) Actions List, (3) Screen Profile/Layout Preview, (4) Tabbed Area (Properties/Toolbox/Assets), (5) Preview Commands.

3. What are the three essential components of ViewApp navigation?

Navigation Model (defines hierarchy), Navigation Item (unique element with visual info and actions), and Navigation Actions (runtime behavior of controls).

4. What is the default name of the root item in the built-in navigation model?

"Home" — it cannot be deleted but can be renamed. "Assets" appears as a top-level item beneath it.

5. What are the four Auto-Fill Content options?

Current only, Current then up, Current up then down, and None.

6. Can you derive a ViewApp instance directly from the $ViewApp base object?

No. You must first derive a template from $ViewApp, configure it, and then derive an instance from the derived template.

7. What type of object must host a ViewApp instance in Deployment view?

A ViewEngine object. The ViewEngine must be deployed before ViewApp instances can be deployed.

8. What happens when automatic update is enabled in preview mode?

Changes saved to the ViewApp are automatically reflected in the preview session without needing to manually click Update.
