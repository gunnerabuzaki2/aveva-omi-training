Introduction to Security in Application Server

The security system in Application Server is a global function that applies to every object in the Galaxy database. It operates as a relationship-based security model connecting users, objects, and the functions of the Galaxy. System administrators can define users and assign the operations they are permitted to perform, with security permissions specified in terms of operations users can carry out using automation objects. The system supports multi-user environments in both development and runtime contexts.

Key Concept: Application Server security is a relationship-based system that ties users to Galaxy objects and functions. Security is a global setting — once enabled, it applies to all objects in the Galaxy database.

Authentication Modes

When Galaxy security is enabled in the System Platform IDE, four authentication modes are available. Each mode determines how user credentials are verified and how access to Galaxy areas and activities is granted.

ModeDescriptionHow It Works
GalaxyUses local Galaxy configuration to authenticate usersSecurity credentials checked and access granted at the Galaxy level
OS User BasedLeverages the operating system's user authenticationSecurity specified at the individual OS user level
OS Group BasedUses OS group-based authentication with user-to-role mappingOS groups mapped to Galaxy security roles; credentials verified at the OS group level
Authentication ProvidersUses AVEVA Identity Manager (AIM) for unified securityIntegrates OS security and Azure Active Directory across nodes and Azure VMs
Tip: OS User and OS Group-based modes add local OS accounts as authorized Galaxy users. With OS Group Based, the local account must exist on each node in the Galaxy for successful authentication across all computers.

Two users are defined by default: Administrator and DefaultUser. These cannot be deleted in an open security setting, and both are associated with default roles.

Security Roles and Permissions

You can create and manage user roles that align with your organization's processes and work-based authorities. Two roles are defined by default: Administrator and Default. Roles are listed with an assigned Access Level — an integer from 0 to 9999 that can be used in graphic animations for security purposes and to configure navigation item security in a ViewApp.

Security Roles configuration — The Administrator role (Access Level 9999) with full permissions, and the Default role (Access Level 0). General permissions include IDE and SMC permissions; Operational permissions are based on Security Groups.

General and Operational Permissions

You can specify two categories of permissions for each role. General permissions relate to application configuration and administrative tasks such as IDE access, importing/exporting, system configuration, deployment, and graphic management. Operational permissions relate to security groups listed on the Security Groups page and control what operations users can perform on objects at runtime — such as acknowledging alarms, modifying attributes, or verifying writes.

By default, both the Administrator and Default roles have all permissions. All users belong to the Default role, and this relationship cannot be changed. Therefore, it is highly recommended that you remove all permissions from the Default role to prevent unintended access.

Important: You cannot modify the General permissions for the role of Administrator. The Administrator role always retains full General permissions.
General Permissions for Industrial Graphics Management — The "Graphic Management Permissions" checkbox is a specific permission unique to industrial graphics within the software.

Security Audit Trail

The Galaxy generates events for user actions such as logging in or logging off, successful and failed attempts to write to attributes, and other activities. Event messages are logged to history blocks or the Alarms and Events database through Historian Server and can be retrieved for analysis and reporting.

ViewApp Security

A ViewApp has built-in security that is integrated with Galaxy security. Industrial graphics can be shown or hidden during runtime based on the currently logged-in user's information. For example, you can use the current logged-in user information as a condition to configure the visibility animation for a specific graphic element — if the condition is true, the graphic displays at runtime. During runtime, ViewApp navigation items can be shown or hidden based on the access level assigned to, and/or user roles selected for, each navigation item.

Key Concept: ViewApp security operates through three mechanisms: (1) Login security via ShowLoginDialog() or TitleBarApp's Login button, (2) Visibility animations using security-related attributes like MyViewApp.Security.LoggedInAccessLevel, and (3) Access Level and User Roles properties on navigation items.

Before implementing security in a ViewApp, security must be enabled for the Galaxy with an authentication mode, and roles and users must be created with proper Operational permissions. ViewApp users must be added to the security system with a username and password, associated with a role and a defined Access Level.

Navigation Item Security Properties

When a navigation item is selected in the ViewApp Editor, two security properties are available in the Security category on the Properties tab: Access Level (integer 0–9999) and User Roles (one or more user roles). If both properties are configured, both conditions must be satisfied for the navigation item to be visible at runtime.

Security ScenarioConfigurationVisibility Requirement
No securityAccess Level = Not Configured; User Roles = Not ConfiguredAlways visible
Access Level onlyAccess Level = Integer <N>; User Roles = Not ConfiguredUser's access level ≥ <N>; any role
User Roles onlyAccess Level = Not Configured; User Roles = Role1, Role2User must belong to at least one specified role
Access Level + User RolesAccess Level = Integer <N>; User Roles = Role1, Role2Access level ≥ <N> AND at least one specified role
Navigation Item Security Properties — The Security section in the Properties tab shows Access Level and User Roles, both defaulting to "Not Configured".

Configuring Access Level

To configure the Access Level property, click the down-arrow to the right of the Access Level property and select an access level from the drop-down list. The list includes access levels assigned to security roles in the Galaxy. Selecting an access level automatically selects all access levels greater than or equal to the one selected. Alternatively, you can click the Access Level field, enter a numerical value, and press Enter.

Access Level dropdown — When "0 (Default)" is selected, all roles with access level ≥ 0 are automatically checked.
Access Level set to 5000 — Only roles with access level ≥ 5000 (Administrator and Plant Supervisors) remain selected.

Configuring User Roles

To configure User Roles, click the down-arrow next to the User Roles property and select one or more user roles. If a parent navigation item is hidden for a logged-in user, all child navigation items are automatically hidden regardless of their own security configurations.

User Roles configuration — Two user roles selected: Cloud\Application Administrators and Cloud\Plant Operators 1.

Security-Related Functions

A typical login interface includes a login button and a logout button. The login button displays an interface for users to enter their username and password. The logout button logs the user off the ViewApp. The following Action Script functions are available:

FunctionDescription
ShowLoginDialog()Shows a login dialog box with fields for username and password
LogOff()Logs a user off of a ViewApp

Single Sign-On (SSO) for ViewApps

If security is enabled with OS User Based or OS Group Based authentication, the current Windows user will automatically log on to the ViewApp when it starts. If the current Windows user is a valid Galaxy user, auto-login succeeds. If the current Windows user is not valid for the Galaxy, auto-login fails and the ViewApp runs without a logged-in user, allowing other users to log on manually.

OMI Apps that use the Single Sign-On feature run in the context of the logged-on ViewApp user. Messages will appear if the current user is not valid for the OMI Apps or does not have sufficient permissions. The SSO feature can be enabled or disabled by setting the AutoLogonCurrentUser attribute to True or False.

Key Concept: Single Sign-On automatically logs the Windows user into the ViewApp at startup. Additional credentials may be needed for third-party applications that don't support standard enterprise authentication — these are managed via the Credentials tab in the Configure Security dialog box.

Knowledge Check

Q1: What are the four authentication modes available when Galaxy security is enabled?

The four modes are: Galaxy (local Galaxy configuration), OS User Based (individual OS user), OS Group Based (OS group mapping), and Authentication Providers (AVEVA Identity Manager with Azure AD integration).

Q2: Why should you remove all permissions from the Default role?

All users belong to the Default role by default, and this relationship cannot be changed. If the Default role retains all permissions, every user would have full access. Removing permissions from Default ensures users only get the permissions assigned through their specific roles.

Q3: What happens if both Access Level and User Roles are configured for a navigation item?

Both conditions must be satisfied — the logged-in user must have an access level equal to or greater than the specified value AND must belong to at least one of the specified user roles.

Q4: Can you modify the General permissions for the Administrator role?

No. The General permissions for the Administrator role cannot be modified — the Administrator always retains full General permissions.

Q5: What are the two Action Script functions for login/logout in ViewApps?

ShowLoginDialog() displays a login dialog box with username and password fields. LogOff() logs the active user off of the ViewApp.
