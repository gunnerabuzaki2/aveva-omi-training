Introduction to Alarms and Events

The alarm and event capabilities in Application Server allow users to automate the detection, notification, historization, and viewing of process alarms and events, as well as system and software alarms and events. Alarms and events are both occurrences in the runtime system, but the system distinguishes between them based on their nature and the response they require.

Key Concept: An alarm represents an abnormal condition requiring immediate user attention (e.g., high temperature, stopped pump). An event is a condition occurrence at a specific point in time (e.g., deploying a new object, a platform coming back online).

The following items are not considered alarms or events: configuration actions within the Galaxy Repository (such as import or checkout), and messages sent to the system logger (OCMCLogger). Note that while certain software events may log a message in addition to generating an event, logger messages themselves are not events.

The Platform as an Alarm Provider

A Platform can act as a single alarm provider, providing alarms to the visualization client's distributed alarm subsystem. A Boolean check box in the Platform automation object indicates whether it subscribes to Areas for alarm and event reporting. The Platform routes all alarms and events for subscribed areas to the distributed alarming infrastructure.

Each alarm generated by Application Server contains the following information:

Alarm DataDescription
TimestampTime of the alarm event
Attribute nameThe attribute that generated the alarm
Common message textHuman-readable alarm message
AreaThe area containing the alarm source
Alarm primitive nameFor example, PV.HiAlarm
New alarm stateFor example, ACK (acknowledged) or RTN (return to normal)
PriorityInteger from 1 to 999
SeverityInteger from 1 to 4 (Critical, High, Medium, Low)

Visualization Clients as Alarm Consumers

Visualization clients consume the alarms and events subscribed by alarm providers. An alarm provider is specified by providing the node name and the name of the provider. Only one alarm provider can exist per node. Clients can be configured with query filters based on subscribed areas to control which alarms are consumed.

Alarm Acknowledgment

Visualization clients can allow operators to acknowledge alarms reported by the Platform. The Platform receives the acknowledgment request and forwards it to the target automation object, where the alarmed attribute is set to acknowledged. Security checks apply during this process, and an optional comment can be entered during acknowledgment.

Alarm Enable/Disable

Alarming on any automation object can be enabled or disabled. The Platform forwards the request to the target object's AlarmMode attribute. Security checks apply to this operation as well.

Alarm Queries

Alarm queries are used within Alarm Clients to retrieve real-time alarm information and event reports from the Galaxy. The fully qualified syntax for an alarm query is:

\\nodename\Galaxy!area!object.attribute

If the Platform alarm provider is on the local computer, you can use the shorter syntax: \Galaxy!area!object.attribute

Alarm QueryReturned Data
\\nodename\Galaxy!areaAll alarms/events from the area, all objects in the area, all subareas, and objects in contained areas
\\nodename\Galaxy!area!areaAll alarms/events from the area object itself only
\\nodename\Galaxy!winplatformAll alarms/events from the Platform object itself
\\nodename\Galaxy!engineAll alarms/events from the engine object (AppEngine or ViewEngine)
\\nodename\Galaxy!dioAll alarms/events from the device integration object itself

Wildcard Queries

The asterisk (*) is a wildcard character that can substitute any characters in the object.attribute part of the alarm query:

Alarm QueryResults
\\nodename\Galaxy!area\object.*All alarms/events from a specific object within an area
\\nodename\Galaxy!area\*.attributeAll alarms/events from all objects for a specific attribute
\\nodename\Galaxy!area\objectprefix*All alarms/events from objects whose name begins with the prefix

Multiple Queries

Alarm Clients can submit multiple queries in one request. This is useful for querying mutually exclusive areas or multiple different nodes simultaneously. For example:

\Galaxy!Area1 \Galaxy!Area2 — retrieves alarms from two non-overlapping areas.

\\NodeNameA\Galaxy!Area1 \\NodeNameB\Galaxy!Area2 — retrieves alarms from different nodes.

Alarm Aggregation

Alarm aggregation provides an efficient way to identify whether any alarms on an object are currently in the InAlarm state, determine the overall status of the most important alarms, and count active alarms by severity and containment level. This makes it possible to follow a trail from one level to the next to find the underlying cause of complex object alarms.

Key Concept: Alarm aggregation works at three levels: Object, Area, and Attribute. It aggregates alarms in three states: UNACK_ALM (unacknowledged active alarm), ACK_ALM (acknowledged active alarm), and UNACK_RTN (unacknowledged return-to-normal). Alarms in the ACK_RTN state are not aggregated. Silenced and shelved alarms are included in aggregation.

Aggregation Attributes

Five read-only attributes provide runtime aggregated alarm status information:

AttributeDescription
<Object>.AlarmMostUrgentInAlarmBoolean — whether there is an active alarm
<Object>.AlarmMostUrgentAckedBoolean — acknowledgment status of the highest-priority alarm
<Object>.AlarmMostUrgentModeEnabled/Silenced/Disabled mode of the highest-priority alarm
<Object>.AlarmMostUrgentSeverityInteger 1–4 (severity) or 0 if no active alarms
<Object>.AlarmMostUrgentShelvedBoolean — shelved status of the highest-priority alarm
<Object>.AlarmCntsBySeverity[]Array — counts of all alarms by severity levels and status

To enable aggregation, check the Aggregate Alarms checkbox in the area object configuration editor. This sets the AlarmAggregationStateCmd attribute to True.

Enable alarm aggregation — Check the "Aggregate Alarms" checkbox in the area object's General tab to enable alarm state aggregation.

Alarm Border Animation

Alarm Border animation shows a highly visible border around a symbol or graphic element when an alarm occurs. The color and fill pattern of the border indicate the severity and current state of the alarm, allowing plant operators to quickly recognize alarm conditions.

Alarm Border animation examples — A pressure gauge with a dark closed-style border and a level indicator with a bright magenta open-style border, both indicating active alarms.

Note: Alarm border animation can be applied to all types of symbols except embedded symbols and nested groups. It can also be applied to graphic elements and grouped elements.

Configuring Alarm Border Animation

Alarm Border animation is configured by selecting "Alarm Border" from the Visualization animations list. The pane contains mutually exclusive fields to set referenced attributes for aggregate or individual alarms. For aggregate alarms, specify an object name in the "Use Standard Alarm-Urgency References" field. For individual alarms, specify dedicated source fields: InAlarm Source, Acked Source, Mode Source, Severity Source, and Shelved Source.

Alarm Border configuration — Use "Me.Alarm.Condition" as the object reference to auto-populate all alarm urgency attributes.

Alarm Border Indicator Icon

The indicator icon appears at the top left corner of the border with alarm severity as a number from 1 to 4. Other indicator images represent alarm suppressed, silenced, and shelved modes. Default images can be replaced with custom images (supported formats: .bmp, .gif, .jpg, .jpeg, .tif, .tiff, .png, .ico, .emf).

Alarm severity configuration — Default indicator images for each severity level (Critical=red, High=green, Medium=teal, Low=purple) and alarm states (Inhibited, Silenced, Shelved).

Shelving Alarms

Alarm shelving allows you to temporarily hide active alarms from displays for a fixed period while they continue to be historized. Shelving is typically used to reduce alarm noise during system modifications or repairs, allowing operators to focus on more urgent alarms.

Shelved alarms differ from silenced alarms in several important ways:

  • Auto-timeout: Shelved alarms automatically unshelve when the configured period expires. You can also manually unshelve them early.
  • Granularity: Shelving applies only to individual alarms, not entire areas or hierarchies. It must be enabled at the area level.
  • Role-based limits: The system enforces role-based restrictions on shelving permissions, severity levels that can be shelved, and the total number of alarms a user can shelve.

The system tracks who shelved the alarm, from what workstation, the reason for shelving, when shelving began, and when it will expire. Alarm shelving is configured per severity level in the Alarms and Events dialog box (Galaxy menu → Configure → Alarms and Events), and can be enabled or disabled per area via the "Shelve Alarms" checkbox in the area object's General tab.

Enable alarm shelving — By default, only Medium and Low severity alarms are selected for shelving at runtime.
Area-level shelving — The "Shelve Alarms" checkbox in the area object's General tab enables or disables shelving for that area.

Knowledge Check

Q1: What is the difference between an alarm and an event?

An alarm represents an abnormal condition requiring immediate user attention (e.g., high temperature). An event is a condition occurrence at a specific time (e.g., deploying a new automation object, a platform coming back online).

Q2: Which alarm states are included in aggregation?

UNACK_ALM (unacknowledged active alarm), ACK_ALM (acknowledged active alarm), and UNACK_RTN (unacknowledged return-to-normal). ACK_RTN is NOT included. Silenced and shelved alarms ARE included.

Q3: What does the alarm border do when a user acknowledges an alarm that is still active?

The alarm border remains around the symbol but stops blinking, indicating the alarm has been acknowledged but is still active.

Q4: What is the key difference between shelved and silenced alarms?

Shelved alarms have an automatic timeout and unshelve when the period expires. Shelving applies only to individual alarms (not entire areas) and is subject to role-based limits. Silenced alarms have no automatic timeout.

Q5: How do you enable alarm aggregation on an area?

Check the "Aggregate Alarms" checkbox in the area object's General tab. This sets the AlarmAggregationStateCmd attribute to True, enabling aggregation for all sub-areas, contained objects, and their descendants.
