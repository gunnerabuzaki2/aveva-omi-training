📖 L04: Screen Profiles and Layouts

Based on Module 2 — Getting Started: Sections 1–3 (pages 2-3 through 2-26)

AVEVA Operations Management Interface (OMI) is the modern visualization layer of the AVEVA System Platform. It leverages both HMI and SCADA systems to deliver a convergence platform for Operational Technology (OT) and Information Technology (IT). Built on the same core ArchestrA technology as System Platform, the OMI visualization engine provides rich, modern user experiences across platforms. This lesson introduces the foundational components — screen profiles, layouts, panes, symbols, OMI Apps, and widgets — that form the framework for building a ViewApp.

1. What is Operations Management Interface?

Source: page 2-3

With Operations Management Interface, applications can be assembled based on a data model, resulting in minimal engineering and increased manageability. An out-of-the-box framework provides native multi-monitor support. OMI is installed as part of Application Server and can coexist on the same computer as InTouch for System Platform.

An OMI application is called a View Application (or simply ViewApp). ViewApps are represented as objects that are created, configured, and deployed from within the System Platform IDE. Once deployed, they can be launched through the AVEVA OMI Application Manager or the AVEVA InTouch Application Manager.

Key Concept: A ViewApp is the fundamental unit of an OMI application. It is created from a derived $ViewApp template, configured with screen profiles and layouts, and deployed to a $ViewEngine host.

Key Features and Capabilities

OMI provides a rich set of features that make it a powerful visualization platform:

Built-in NavigationA default navigation model based on the Galaxy's plant model, which can be fully customized.

2. Visualization Components — System Objects

Source: page 2-5

Two system objects in the System Platform IDE are essential for creating OMI ViewApps: $ViewApp and $ViewEngine. These form the backbone of the visualization architecture.

System ObjectPurpose
$ViewApp Base system object from which a ViewApp template is derived. Configuration happens in the derived template; an instance is created from it and deployed to run the ViewApp.
$ViewEngine Base system object from which an instance is created to host ViewApp and InTouchViewApp objects. Supports deployment, startup, and shutdown. Does not support redundancy. Must be assigned and deployed to a WinPlatform object.

The $ViewEngine object has several important capabilities: it can host and execute one or more ViewApp and InTouchViewApp objects (with its scan rate determining the scan rate of all hosted objects), determine whether ViewApps run in read/write or read-only mode, include custom attributes and scripts, contain runtime diagnostic attributes that can be monitored, alarmed, and historized, and specify the Historian connection for historization.

The visualization component hierarchy: a ViewApp runs on a ViewEngine, which is deployed to a WinPlatform on top of the Bootstrap layer.
Key Concept: The ViewEngine is the runtime host for ViewApps. It must be deployed to a WinPlatform before any ViewApp instances can be deployed. Think of the ViewEngine as the "player" that executes the ViewApp "content."

3. Screen Profiles, Layouts, and Panes

Source: page 2-6

Three framework components work together to define how a ViewApp appears on screen: screen profiles, layouts, and panes. These are created before you build the ViewApp itself.

Screen Profiles

A screen profile defines the physical characteristics of one or more client workstation monitors that will show a running ViewApp, including how the screens are arranged with respect to one another. Each screen in the profile represents a physical monitor, and you specify details like resolution, orientation, and physical arrangement.

Layouts

A layout consists of one or more rectangular areas of content for a screen, and determines how runtime information is displayed in a ViewApp. The rectangular areas within the layout are called panes. The layout controls how panes are positioned on the screen, as well as other visual characteristics and runtime behaviors.

Panes

A pane is a subdivision of a layout and serves as a container for content to be displayed at runtime. Each pane can be configured to hold zero, one, or more content items — symbols, navigation trees, another layout, camera feeds, and more. A pane can encompass an entire screen or only a portion of a screen.

How a screen profile maps to physical monitors: the profile defines the monitor configuration, and the ViewApp content fills the screens accordingly.
The layout and pane creation workflow: create a layout, add panes, populate panes with content, and the layout appears in a running ViewApp.
Key Concept: Screen profiles define the hardware (monitors), layouts define the screen structure (panes), and panes hold the actual content (symbols, apps, controls). You build them in this order: screen profile → layout → panes → content.

4. Symbols

Source: page 2-7

Symbols from the Industrial Graphic and Situational Awareness libraries are the visual building blocks of a ViewApp. You create them with the Graphic Editor and use them as content within ViewApps. These can be pre-made symbols from the built-in libraries or custom assets created specifically for your application.

Other technologies like Windows Forms and HTML5 can also be used. A set of default Apps is included in the Galaxy, each performing a specific function.

6. Widgets

Source: page 2-8

Widgets are mini-apps that run on ViewApps. They generally display data, and can be interactive and resizable. Widgets typically fall into three categories:

  • Information widgets: Display essential information and track changes over time (e.g., weather widgets, clock widgets).
  • Collection widgets: Display a collection of same-type elements for browsing and detail-viewing workflows.
  • Control widgets: Trigger common functions directly from a running ViewApp without opening an App first (e.g., music play/pause/skip).

Six widgets are provided out-of-the-box: Carousel, Datagrid, Map_App, QRCode_Scanner, Teamwork, and Web_Browser.

7. External Content

Source: page 2-9

External content refers to media content outside of a Galaxy — images, videos, documents, and websites — that can be linked to a ViewApp and displayed at runtime. An external content item identifies the path to the external media. These items can be linked directly to a layout pane or automatically filled through navigation at runtime, potentially replacing some out-of-the-box Apps like DocViewerApp, ImageViewerApp, and PDFViewerApp.

8. Workflow for Building a ViewApp

Source: page 2-9

The general steps for creating an OMI ViewApp follow a structured workflow:

  1. Design standards: Create reusable graphics and object templates that serve as the foundation for ViewApps.
  2. Build a screen profile and layout: Create a screen profile for your monitor configuration, then create a layout to host content.
  3. Build the ViewApp: Derive a ViewApp template from $ViewApp, initialize it with a screen profile and layout, and add content items.
  4. Preview the ViewApp: Test in a runtime preview session before deploying or redeploying.
  5. Deploy the ViewApp: Create a runtime instance from the derived template and host it on a deployed ViewEngine.
  6. Run the ViewApp: Launch from the AVEVA OMI Application Manager or AVEVA Application Manager.

9. Screen Profiles — In Detail

Source: pages 2-11 to 2-15

A screen profile sets up the physical characteristics and layout of client workstation monitors for running a ViewApp. Each screen entry in the profile corresponds to a physical monitor, and you need to specify the number of screens, their resolution, orientation, physical arrangement, primary screen designation, and whether any screens are touch-enabled.

Tip: You can create a screen profile before setting up physical monitors. For testing a multi-screen ViewApp, start with a single monitor — the system will open multiple separate windows to represent all screens defined in the profile.
Multiple workstations can share the same screen profile, or a single workstation can use a custom multi-screen profile. Each screen profile defines resolution, layout assignment, and arrangement.

Screen Profile Editor

You create screen profiles from the Galaxy menu (New → Screen Profile) or from the Graphics tab by right-clicking the Galaxy name or a toolset. The first screen profile receives the default name ScreenProfile_001. Double-click a screen profile to open it in the Screen Profile Editor.

The Screen Profile Editor: (1) screen preview area, (2) zoom controls, (3) add-screen button, (4) screen properties panel showing name, orientation, resolution, and taskbar settings.

Managing Screens

A new screen profile starts with one screen named "Primary," which is the default primary screen. You can add up to 20 screens total. New screens are added to the right of the selected screen and are duplicates of the selected screen's configuration. The primary screen cannot be deleted but can be reassigned by clicking the empty blue checkmark on another screen.

Screens are arranged using drag-and-drop, subject to these rules: all screens must touch at least one other screen, at least one corner must align with another screen's corner, screens must not overlap, and the maximum is 20 screens per profile.

Screen Properties

PropertyDescription
NameUnique name for the screen within the profile.
OrientationLandscape or Portrait. Changing orientation swaps width and height values.
ResolutionWidth and height of the screen in pixels.
Aspect RatioAutomatically calculated from the resolution.
Has TaskbarIndicates whether the screen has a taskbar, which affects layout sizing.
Taskbar PositionAvailable if Has Taskbar is checked. Top, Bottom, Left, or Right.
Is TouchscreenEnables touch gesture support and Touch Lock Level configuration.
Touch Lock LevelNone, Two Handed Operation, or Commands Locked. Controls how touch gestures are restricted at runtime.
Touch Lock PositionBottom Left or Bottom Right. Controls placement of the touch lock icon.

Touch Lock Levels

For ViewApps running on touch screens, three lock levels control touch gesture behavior:

  • None: Touch lock is disabled; full touch interaction is available.
  • Two Handed Operation: A touch lock icon appears. Single-finger content selection is blocked, but two- and three-finger pane gestures are allowed. Pressing the icon temporarily unlocks all gestures.
  • Commands Locked: A touch lock icon appears. All touch gestures are blocked when locked; mouse and keyboard still work. Pressing the icon unlocks touch interaction.

10. Layouts and Panes — In Detail

Source: pages 2-17 to 2-25

A layout defines the organization of rectangular areas (panes) on a screen. Layouts also define the behavior and types of interactions supported by each pane, such as panning, zooming, or resizing at runtime.

A simple two-pane layout: Pane1 (default) on the left and Pane2 on the right. The "+" buttons around the frame allow adding new panes.
A four-pane layout with Pane4 as a slide-in pane on the left. Slide-in panes are hidden at runtime until the user activates them.

Creating a Layout

You create layouts from the Galaxy menu (New → Layout) or from the Graphics tab. The first layout receives the default name Layout_001. Double-click a layout to open it in the Layout Editor, which provides two pages: a Build page for visual editing and a Script page for layout scripts.

The Layout Editor supports two modes:

  • Fixed layout mode (default): For ViewApps running exclusively on desktop monitors.
  • Responsive layout mode: For ViewApps viewed on a mix of devices (phones, tablets, monitors).
Warning: Once you begin adding and configuring content, switching between fixed and responsive layout modes will clear all panes and any unsaved configuration changes will be lost. Unsaved layout scripts are not affected.
The Layout Editor on the Build page: (1) Build/Script selector, (2) Fixed/Responsive toggle, (3) Actions list, (4) layout preview area, (5) zoom/pan controls, (6) tabbed area with Properties, Toolbox, and Assets tabs.

Adding and Managing Panes

Initially, a layout contains one pane named "Pane1" set as the default pane. Use the Add New Pane control in the center of a selected pane to add panes above, below, to the left, or to the right. You can also right-click a pane for options: Add Pane, Extend Pane, Rename Pane, Set as Default Pane, and Delete Pane.

The Add New Pane control appears in the center of a selected pane. Click different extensions to add panes in any direction.

Default Pane and Slide-In Panes

Every layout has one default pane — the pane most likely to display changed content at runtime, labeled "(default)." You can change which pane is the default by selecting a pane, clicking the vertical ellipsis (⋮), and choosing "Set as Default Pane."

Slide-in panes are attached to the border of the layout window and hidden at runtime until the user clicks a corresponding button on the ViewApp window border. A slide-in pane at the top or bottom spans the full width; one on the left or right spans the full height. You can resize them by dragging their edge, but slide-in panes cannot be set as the default pane.

Assigning Content to Panes

Content is assigned to panes from the Toolbox and Assets tabs on the right side of the Layout Editor. You can add symbols, OMI Apps, layouts, external content, or widgets by dragging them from the preview area into a pane. When a pane holds a single content item, a preview appears in the Layout Editor.

Content Types and Auto-Fill

Content types are a classification technique for determining what content can be shown in a pane at runtime. You set the Content Type(s) property for a pane in the Layout Editor. The default is "None," which disables Auto-Fill.

Auto-Fill is a navigation behavior that automatically places content in panes at runtime without manual configuration. It requires two things: the layout pane must have "Use For Auto-Fill" enabled, and the navigation item must have "Auto-Fill Content" configured.

Key Concept: Content types match content items (symbols, layouts, external content) to panes at runtime. When a content item's type matches a pane's configured types, the content automatically fills that pane via Auto-Fill.

Predefined content types include Overview, Detailed, Faceplate, Alarm Banner, Alarm Summary, Real Time Trend, History Trend, Map, Camera, Docs, and Menu. There are also 25 user-defined types (User_Defined_1 through User_Defined_25) available for custom content classifications.

Quiz

Test Your Knowledge

1. What is the name of an AVEVA OMI application?

A ViewApp (View Application). ViewApps are created, configured, and deployed from the System Platform IDE.

2. What two system objects are essential for creating OMI ViewApps?

$ViewApp (the base template for ViewApps) and $ViewEngine (the runtime host that executes ViewApps).

3. What must the ViewEngine object be assigned to before it can be deployed?

A WinPlatform object. The ViewEngine must be assigned and deployed to a WinPlatform before ViewApp instances can be deployed.

4. What is the difference between a screen profile, a layout, and a pane?

A screen profile defines the physical monitor configuration. A layout defines how a screen is divided into rectangular areas. A pane is a subdivision of a layout that serves as a container for content.

5. What happens if you switch between fixed and responsive layout modes after adding content?

All panes are cleared and any unsaved configuration changes are lost. Unsaved layout scripts are not affected.

6. Can a slide-in pane be set as the default pane?

No. Slide-in panes cannot be set as the default pane for the layout.

7. What is the maximum number of screens in a screen profile?

20 screens.

8. What are the three Touch Lock Level settings?

None (full touch interaction), Two Handed Operation (blocks single-finger selection, allows two/three-finger gestures), and Commands Locked (blocks all touch gestures, mouse/keyboard still work).
