🧪 L31: Lab 15 – Creating and Using Custom ViewApp Namespaces and Attributes

Module 7 — ViewApp Namespaces | Lab 15

Pages 413–432

Introduction

In this lab, you will create a sub-toolset under Training on the Graphics tab for storing ViewApp namespaces. You will create a custom ViewApp namespace named Sources and configure it in the ViewApp Namespace Editor. In the editor, you will add three ViewApp attributes, including one named URL1 for providing the URL for the WebBrowser App.

You will then create a symbol with two buttons and a text box. The buttons will be configured to display a homepage and a local HTML file in a running ViewApp. The text box can be used to navigate to other locations, such as other web sites. You will also create a new layout with the symbol in one pane and the WebBrowser App in another.

The running ViewApp navigated to WebBrowser, showing the BrowserURLEntry symbol at the top (Homepage and HTML buttons with a URL text box containing "www.aveva.com") and the WebBrowser App below displaying the AVEVA website.
Objectives:
  • Create custom ViewApp namespaces and attributes
  • Use custom ViewApp namespaces and attributes in symbols and Apps
  • Use the WebBrowser App in a ViewApp
  • Override App properties using references

Create a New Graphic Toolset and a ViewApp Namespace

Step 1: On the Graphics tab, create a new graphic toolset under the Training toolset and name it ViewApp Namespaces.

Key Concept — Custom ViewApp Namespaces: Custom namespaces let you define your own attributes (like URL1, HomeAddress, File_HTML) that store application-specific data within the ViewApp. These attributes can be referenced by animations, bound to App properties, and updated at runtime — providing a flexible way to share state between symbols and Apps.
The Graphics tab showing the newly created "Viewapp Namespaces" folder under the Training toolset, alongside existing symbols and the Widgets folder.

Step 2: Right-click the Training \ ViewApp Namespaces toolset and select New | ViewApp Namespace. A new ViewApp namespace with the default name is created.

The right-click context menu on the Viewapp Namespaces toolset showing the New submenu with "ViewApp Namespace (Ctrl+Shift+V)" highlighted for selection.

Step 3: Rename the new ViewApp namespace to Sources.

The Graphics tab with the Viewapp Namespaces folder expanded, showing the renamed "Sources" namespace with the green AVEVA icon.

The Sources ViewApp Namespace Editor in its empty state — no attributes defined yet. The Add (+), Duplicate, and Remove buttons are visible in the toolbar.

Step 5: Click Add to add a new ViewApp attribute. A new attribute with the default name is added to the list.

The Add button highlighted with a red arrow and tooltip "Add ViewApp Attribute" — clicking this creates a new attribute row in the editor.

Add User Input Animation and Create Layout

Step 27: Double-click TextBox1 and add a User Input animation.

Key Concept — User Input Animation: The User Input animation allows an operator to type a value into a text box at runtime. It writes the entered value to a specified namespace attribute (in this case, MyViewApp.Sources.URL1), which other symbols and Apps then read — enabling dynamic user-driven navigation to different web pages.

Step 28: Configure the animation:

PropertyValue
StatesString
Reference (String)MyViewApp.Sources.URL1
Message to UserEnter a valid URL to browse
The Edit Animations dialog for TextBox1 showing the User Input animation configured with String state, reference to MyViewApp.Sources.URL1, and the message "Enter a valid URL to browse".

Step 29: Click OK to close the Edit Animations dialog box.

Configure WebBrowser Address Property Binding

Step 39: With Pane1 selected, on the Properties tab for the Browser \\ Address property, click the down-arrow to the left of the field. A drop-down list appears.

Key Concept — WebBrowser Address Binding (In vs. Constant): The WebBrowser's Address property must be set to In binding mode (Dynamic) instead of the default Constant mode. This allows the WebBrowser to read its URL from the ViewApp namespace attribute at runtime — so when buttons or the text box update the URL1 attribute, the browser automatically navigates to the new URL.
The Properties tab for the WebBrowser control showing the Browser section with the Address field highlighted. The down-arrow next to the "C" (Constant) icon reveals binding mode options.
The binding mode dropdown showing options: Constant, In, Out, InOut, and Reset. Selecting "In" changes the binding mode to Dynamic.

Step 40: From the drop-down list, select In. This changes the binding mode to Dynamic. The binding mode icon changes to a left-pointing arrow, indicating In-only binding — the property will be overridden by the specified expression or reference.

Step 41: Enter the following for the Browser \ Address property reference: MyViewApp.Sources.URL1

The Browser Address field now shows the In-binding icon (left-pointing arrow) with the value "MyViewApp.Sources.URL1" — the WebBrowser will read its URL from the custom namespace attribute.
The complete Properties panel for WebBrowser1 showing General (Name, Content, Pane), Browser (Address bound to MyViewApp.Sources.URL1 with In-binding), and Event Handlers sections.
Important: The binding mode must be set to In (not Constant) for the WebBrowser to dynamically read the URL from the ViewApp namespace attribute. If left as Constant, the address will be static and will not update when buttons are clicked.

Knowledge Check

Q1: Why is the WebBrowser's Address binding mode changed from Constant to In?

Changing the binding to "In" (Dynamic) allows the WebBrowser to read the URL from the ViewApp namespace attribute at runtime. This means when buttons or the text box update the URL1 attribute, the WebBrowser automatically navigates to the new URL.

Q2: What are the three custom attributes created in the Sources namespace?

URL1 (String, initial value "www.aveva.com", Retentive) — the current URL displayed by the browser; HomeAddress (String, "https://www.aveva.com", Readonly) — the homepage address; File_HTML (String, file path, Readonly) — the local HTML file source.

Q3: What is the purpose of the User Input animation on TextBox1?

The User Input animation allows the operator to type a URL into the text box at runtime. It writes the entered value to MyViewApp.Sources.URL1, which the WebBrowser App reads via its In-binding, enabling dynamic web navigation.

Q4: Why must all objects be closed and checked in before the next lab?

The next lab (Lab 16) covers ViewApp security configuration. All objects must be closed and checked in so that the Galaxy is in a clean, committed state before security settings are applied.
