Module 9 — Alarms and Events Visualization, Section 2

Pages 531–542

Introduction

Application Server handles the real-time reporting of alarms and provides clients for viewing current alarms. It can also detect events and report them to clients as recent events. You can create ViewApps that include a display for alarms and events, allowing operators to view and manage both current alarms and recent events. Current alarms require immediate attention, and operators can acknowledge them directly within the ViewApp. Recent events can be reviewed for analysis and reporting purposes.

AlarmApp is an out-of-the-box OMI App provided to display alarms and events in ViewApps. Using the AlarmApp, an operator can view current alarms and recent events, acknowledge current alarms, query and filter alarms and events, and more.

The AlarmApp

The AlarmApp is an OMI App that you can place in your ViewApps to show live and historical alarms and events. At runtime, the pane configured with the AlarmApp shows alarms and events in tabular form, with each row representing a single alarm or event. The background color of a row for an alarm indicates the severity and current state of the alarm.

You can configure the AlarmApp to show current alarms, recent alarms and events, historical alarms, historical events, or historical alarms and events. You can specify queries to subscribe alarms and events from the alarm provider, and you can specify conditions to filter which alarms and events are displayed.

To use the AlarmApp in a ViewApp, you must assign it to a layout pane. You can do this from either the Layout Editor or the ViewApp Editor. Select the AlarmApp from the Toolbox tab, and from the preview area, drag and drop the AlarmControl to the target pane.

Configure the AlarmApp

You can configure AlarmControl properties from either the Layout Editor or the ViewApp Editor. When configuring the AlarmApp, properties appear on the Properties tab when the pane that contains the AlarmControl is selected.

General Properties

PropertyDescription
NameShows the name of the content. By default, it is auto-generated based on the assigned control. It is editable.
ContentShows the name of the AlarmControl assigned as the content. By default, displays as AlarmApp.AlarmControl.
PaneShows the pane that contains the AlarmControl.
Properties panel displaying the General section and collapsible property groups for the AlarmControl, including Run-Time Behavior, Configuration, Run-Time Values, Appearance, Command Bar, Alarm Mode, Area Hierarchy, and .NET AlarmClientControl groups

Run-Time Behavior Properties

Run-Time Behavior properties provide configurations for AlarmApp runtime behaviors, such as how the AlarmApp works within the context of the current area, whether column headings are shown, how alarms can be grouped, and whether signatures are required to acknowledge alarms. Key properties include:

PropertyDescription
Follow Current ContextDetermines if the AlarmApp participates in the context of the current area. When enabled, the alarm query automatically changes to \\Galaxy\<CurrentArea> at runtime.
Only Show Alarms on Current AssetWhen Follow Current Context is enabled, this determines if the AlarmApp shows aggregated alarms for all assets in the area, or only alarms for the currently-selected asset.
Navigate On Double ClickDetermines if double-clicking an alarm navigates to the asset that generated it. Enabled by default.

Configuration, Run-Time Values, and Appearance Properties

The Configuration property provides an ellipsis button to open the configuration editor for the AlarmApp, where you can set alarm mode, colors, columns, queries, and more.

Run-Time Values properties include read-only properties like Selected Row Tagname and Selected Row Asset, which can be bound to references (such as ViewApp namespace attributes) to retrieve the tagname or asset of a manually selected row at runtime.

Appearance properties configure the background and foreground colors for the AlarmControl's command bar, buttons, and area hierarchy display. These colors do not apply to the alarm grid itself.

Command Bar Properties

Command Bar properties control the alarm grid's command bar. Show Command Bar determines if the command bar is shown above the alarm grid at runtime. Available buttons include Navigate, Ack Selected, Ack All, Ack Visible, Requery, Connect, and Disconnect. The Show Command Bar property can bind to a reference (such as a ViewApp namespace attribute) to dynamically show or hide the command bar at runtime.

Alarm Mode Properties

PropertyDescription
Alarm QueryRead/write property to get or set an alarm query dynamically at runtime. Can bind to a ViewApp namespace attribute or application object attribute. Overrides the query in the Configuration dialog.
Client ModeRead/write integer value (1–5) that determines the type of alarm data shown in the AlarmApp. Can bind to a namespace reference or attribute.

Area Hierarchy Properties

Area Hierarchy properties determine if the area hierarchy tree is shown or hidden in the AlarmApp at runtime, and include font settings for the tree if shown.

PropertyDefaultDescription
Show Area HierarchyDisabledToggles the area hierarchy tree view within the AlarmApp. Can bind to a reference for dynamic toggling at runtime.
Show Alarm IndicatorEnabledDetermines if an alarm indicator appears beneath an area name in the area hierarchy.
Binding the Show Command Bar property to a dynamic reference — here the property is bound to MyContent.AlarmAppConfig1.ShowCommands, a public custom property on a symbol in the same layout

Binding AlarmApp Run-Time Properties

The AlarmApp includes runtime properties exposed as configurable properties, which determine the visual aspects or functional behaviors of the AlarmApp during runtime. Configuring some of these properties also applies to the configurations available in the Configuration Editor, such as Alarm Query, Client Mode, and Requires Ack Signature.

Most properties include a list of binding options to configure a static or dynamic value. You can select a dynamic binding option and configure the value by specifying a reference — for example, a reference to an attribute defined in a custom ViewApp namespace, or a public custom property of a symbol assigned in the same layout. With dynamic binding, these properties can change at runtime based on the reference values.

Example: The Show Command Bar property can bind to MyContent.AlarmAppConfig1.ShowCommands, where ShowCommands is a public custom property defined in a symbol named AlarmAppConfig, assigned to the same layout as the AlarmControl.

Event Handlers

The Event Handlers category on the Properties tab lists all available events for the AlarmControl, and you can create an event handler script by selecting an event from the list. The underlying control includes a set of public events identified by the AlarmClientControl prefix in their names. After creating an event handler script, you can navigate to the Script page to edit it by clicking the ellipsis button to the right of the added script.

The Event Handlers section on the Properties tab, listing available AlarmClientControl events (such as NewAlarm, FocusChanged, etc.) — the count shows 191 available events

The AlarmApp Configuration Editor — Alarm Mode tab showing Client Mode set to "Current Alarms" and the Alarm Query field with the query string

Alarm Mode Page

Options on the Alarm Mode page depend on which Client Mode is selected. If Current Alarms or Recent Alarms and Events is selected, fields for specifying a default alarm query and default acknowledgment comment appear. A default alarm query must include Provider and Group as criteria. If Historical Alarms, Historical Events, or Historical Alarms and Events is selected, fields for database connectivity and other settings appear instead.

Alarm Queries

The AlarmApp supports standard Galaxy alarm query formats. Examples include:

Query FormatDescription
\galaxy!areaLocal Galaxy query for a specified area
\\node\galaxy!areaRemote Galaxy query targeting a specific node

Query syntax is the same for both Current Alarms and Recent Alarms and Events. Queries for Historical Alarms, Historical Events, and Historical Alarms and Events follow SQL Server database query rules instead.

Queries and Filters Page

You can use the Queries and Filters page to create queries and filters to show alarms and events as needed. The default query specified on the Alarm Mode page is displayed as the "Default" query — it cannot be deleted but can be customized. Queries and filters configured on this page are available from a list of preconfigured queries and filters at runtime. You can also create new queries and filters at runtime.

The Queries and Filters page — the default query is configured to display current alarms from Line1 and Line2 areas using OR/AND logic

Runtime Behavior Page

You use the Runtime Behavior page to configure the appearance and behavior of the AlarmApp at runtime. This includes toggling options such as Show Heading, Show Grid, and Show Status Bar to control the visual appearance, and behavior options like Disable File Browsing and Overwrite Design Time Queries and Filters. You can also configure whether signatures are required for shelving and acknowledging alarms.

The Run-Time Behavior page — left pane shows appearance checkboxes (Show Heading, Show Status Bar, Query on Startup, etc.), right pane shows Context Menu command options

Manage Alarms at Runtime

The AlarmApp provides comprehensive alarm management capabilities at runtime, including acknowledging, shelving, hiding, querying, filtering, and sorting alarms.

Acknowledge Alarms

Acknowledging current alarms ensures that an operator is aware of a temporary alarm state change. To acknowledge alarms through the AlarmApp, the Client Mode must be configured as Current Alarms. You can acknowledge one or more selected alarms, all alarms (including those not visible due to display limitations), all visible alarms, or all alarms with common values such as alarm priority. An optional comment can be entered and will be logged with the acknowledgment record. When security is enabled, alarms can only be acknowledged by operators with proper authorization.

Shelve and Unshelve Alarms

Shelving alarms allows operators to temporarily suppress them for a fixed period. Shelved alarms are filtered from the alarm grid and can only be shown when a query or filter configured to show shelved alarms is selected at runtime. Shelving/unshelving only applies when the Client Mode is configured as Current Alarms. Shelved alarms can be manually unshelved before the shelving duration expires by first creating a query or filter to show them, then using the context menu to unshelve.

Ack Alarm dialog (left) with Comment, Username, Password, and Domain fields for signed acknowledgment; Shelve Comment dialog (right) with Duration, Reason, and credential fields

Query and Filter at Runtime

Alarms can be queried and filtered at runtime using pre-defined queries and filters. You select an existing query to subscribe to matching alarms, then apply a filter to narrow the results without re-running the query. Queries and filters can also be created at runtime. If Overwrite Design Time Queries and Filters is enabled, runtime queries always take precedence. If Disable File Browsing is unchecked, queries and filters can be exported to and imported from XML files.

Hide and Sort Alarms

Alarms can be hidden or unhidden in the AlarmApp grid. Hiding alarms causes the AlarmApp to ignore them, but the actual alarm generation is completely unaffected — alarm records are still logged into alarm history. The AlarmApp supports sorting for up to four columns at design time, and operators can configure additional column sorting at runtime by clicking column headers. Changes to sorting at runtime are lost when returning to design time.

Freeze and Unfreeze

You can freeze the alarm grid to prevent updates. For example, if new alarms occur while the grid is frozen, they only appear after unfreezing. If a duration is configured, the grid automatically unfreezes after that period. The grid also automatically unfreezes when the Alarm Mode, Alarm Query, or Query Filter changes.

The Ack Alarm dialog for signed acknowledgment — when alarms fall within the configured priority range, the comment is prefixed with "Signed ACK -" indicating a signed acknowledgment

Key Takeaways

  • The AlarmApp is the primary OMI App for displaying and managing alarms and events in ViewApps
  • Assign it to a layout pane via drag-and-drop from the Toolbox tab
  • Properties control runtime behavior, appearance, alarm queries, and command bar visibility
  • Most properties support dynamic binding to namespace attributes or symbol properties for runtime flexibility
  • The Configuration Editor provides six tabs for alarm mode, colors, columns, queries, time, and runtime behavior
  • Alarms can be acknowledged, shelved, hidden, queried, filtered, and sorted at runtime

Knowledge Check

Q1: What is the AlarmApp?

The AlarmApp is an out-of-the-box OMI App used to display live and historical alarms and events in ViewApps. It shows alarms in tabular form with row background colors indicating severity and state.

Q2: How do you assign the AlarmApp to a ViewApp?

Select the AlarmApp from the Toolbox tab in the Layout Editor or ViewApp Editor, then drag and drop the AlarmControl to the desired layout pane.

Q3: What does the "Follow Current Context" property do?

When enabled, the AlarmApp participates in the context of the current area, and the alarm query automatically changes to show alarms for the currently-navigated area at runtime.

Q4: What are the three main alarm management operations available at runtime?

Acknowledging alarms, shelving/unshelving alarms, and hiding/unhiding alarms. Queries, filters, and sorting are also available.

Q5: What does the "Show Command Bar" property control?

It determines whether the command bar (with buttons like Navigate, Ack Selected, Ack All, Requery, Connect, Disconnect) appears above the alarm grid at runtime. It can be bound to a reference for dynamic toggling.
