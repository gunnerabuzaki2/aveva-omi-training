🔬 L02: Lab 1 – Creating and Deploying a Galaxy

Source: AVEVA™ Operations Management Interface 2023 — Module 1: Introduction

Pages 17–32

Lab Introduction

In this lab, you will create a Galaxy based on the Blank Galaxy template and connect to it using the System Platform IDE. Then you will import some prebuilt automation objects into the Galaxy and configure some of them — for example, you will configure a WinPlatform to work in a multi-node environment and an AppEngine to historize data to Historian. Once the Galaxy is created and configured, you will deploy it and use Object Viewer to verify that attributes are receiving data in runtime.

The Galaxy you create in this lab will be used for the other labs in this course.

Lab Objectives: Upon completion, you will be able to:
  • Create a Galaxy
  • Import objects into a Galaxy
  • Deploy objects
  • Verify data using Object Viewer

Part 1 — Create a Galaxy

First, you will create a Galaxy based on the Blank Galaxy template and connect to it.

Step 1: Open the System Platform IDE. The Galaxies dialog box appears.
The System Platform IDE startup Galaxies dialog. The left sidebar shows navigation tabs (Home, New, Open, License). The main area displays server connection options and new galaxy templates — including Default Galaxy, Blank Galaxy, InTouch Base, and Reactor Demo.
Step 2: In the New area, select the Blank Galaxy template. The Create Galaxy dialog box appears.
Note: Be sure to select the Blank Galaxy template. This creates a Galaxy containing only base template objects. You will import prebuilt objects into it in a later step.
The Create Galaxy dialog box. The Galaxy name field is empty with a blue underline, the template is set to "Default_EMPTY.cab" (Blank Galaxy), and the "Connect to this Galaxy" checkbox is checked by default. The Create button is grayed out until a name is entered.
Step 3: In the Galaxy name field, enter TrainingGalaxy. Notice that the Connect to this Galaxy option is enabled by default.
Step 4: Click Create. The Create Galaxy dialog box shows the progress of the Galaxy creation process, as well as informational message details. This will take a few moments.
Step 3: The Create Galaxy dialog with "TrainingGalaxy" entered in the name field. The template shows "Default_EMPTY.cab" and the "Connect to this Galaxy" checkbox is checked.
Step 4: The Galaxy creation progress dialog showing "Creating galaxy TrainingGalaxy on S00ENG..." with a progress bar at approximately 35%. The details section shows the creation steps being executed.
Important: Review the messages shown in the Create Galaxy dialog box to make sure no errors occurred during creation.
Step 5: When the Galaxy creation progress indicates Successfully created TrainingGalaxy, the Close button becomes enabled. Click Close. After a few moments, the Galaxy opens in the System Platform IDE.
The completion dialog confirming "Successfully created TrainingGalaxy on S00ENG." The details list shows steps including restore file repository, restore global data, distribute Galaxy data, and successfully deployed ASB services.
The System Platform IDE after Galaxy creation. The window title reads "TrainingGalaxy on S00ENG – AVEVA System Platform IDE." The ribbon toolbar shows file operations, creation tools (Symbol, Template, Instance, Screen), and action buttons (Deploy, Check-in, Runtime). The left pane shows the Templates tree.

Part 2 — Import Objects

Next, you will import some prebuilt objects into the Galaxy.

Step 6: On the ribbon, select the Galaxy menu, and then select Import > Objects > From package. The Import Objects from package dialog box appears.
Note: Selecting the Galaxy menu opens the System Platform IDE backstage view.
The System Platform IDE backstage view with the Import section highlighted. The left sidebar shows Home, New, Open, User, AVEVA Connect, Import, Export, Configure, and Close options.
The Import Objects from package file picker dialog. Navigate to the training files directory to locate the package file.
Step 7: Navigate to C:\Training, select the Lab 1 - Galaxy_Prebuilt.aaPKG file, and then click Open. The Import preferences dialog box appears.
The file picker in C:\Training showing the Lab 1 - Galaxy_Prebuilt.aaPKG package file selected (19,355 KB), alongside other lab package files.
The Import preferences dialog with conflict resolution settings: overwrite objects with higher imported version, migrate base templates with different revisions, skip objects with different codebases, and never overwrite unprotected objects with protected ones.
Step 8: Leave the default import preferences and click Import. The Import Objects from package dialog box shows the progress of the import process.
Important: Review the messages shown in the Import Objects from package dialog box to make sure no errors occurred.
Step 9: When the import progress indicates Import completed, click Close.

Note: Your instructor will provide the node name for this computer. In the example below, S00PROD is shown as the computer name.

Step 14: Save and close the PROD_Platform1 editor. The Check-in dialog box appears.
Step 15: Enter a comment and click Check-in.
Note: Entering a comment in the Check-in dialog box is optional, but recommended for tracking changes.
The Check-in dialog box with the comment "Configured network address for Prod_Platform1" entered. The "Do not prompt for check-in comments in the future" checkbox is unchecked.

In the following steps, you will configure the Historian connection for an AppEngine object named AppEngine1.

Note: Your instructor will provide the node name. In the example, S00ENG is shown as the Historian node name.
Step 18: Save and close the AppEngine1 editor, and check in the object.
The Object Editor for AppEngine1 with the General tab active. Key settings include Engine startup type (Auto), Scan period (500 ms), and the Historian section with "Enable storage to historian" and "Enable Tag Hierarchy" both checked. The Historian node is set to "S00ENG".

The "Deploy 104 objects" dialog box showing deployment defaults (Cascade deploy checked), deployed objects settings (Skip selected, Force off scan checked), undeployed objects (Deploy new objects checked), and initial scan state (On scan selected).
Step 20: Leave the default deployment settings and click Deploy. The Deploying Objects dialog box shows the progress of the deployment process. This will take a few moments.
Important: Review the messages shown in the Deploying Objects dialog box to make sure no errors occurred during deployment.

The Deploying Objects dialog showing "Deploy complete" status. The details log shows deploying 1 OINetwork object (ProdPLC1), 8 Areas, 92 Automation Objects (starting with Agitator_001), placing 101 objects on scan, and confirming "Deploy Completed: 103 objects out of 103 selected."

End of Lab 1. You have successfully created a Galaxy from the Blank Galaxy template, imported prebuilt automation objects, configured the platform and engine for Historian, deployed to runtime, and verified data flow using Object Viewer. This Galaxy will be used for all subsequent labs in this course.

🧠 Knowledge Check

Q1: Why do we use the Blank Galaxy template instead of the Default Galaxy?

The Blank Galaxy template creates a Galaxy containing only base template objects, which gives us a clean starting point. We then import specific prebuilt objects for the course labs, ensuring a controlled and known configuration.

Q2: What must be configured before deploying the Galaxy?

Two key configurations are needed: (1) the WinPlatform (PROD_Platform1) must have its Network address set to the target deployment computer, and (2) the AppEngine (AppEngine1) must have its Historian field configured with the Historian node name.

Q3: How do you verify that a deployed Galaxy is receiving data at runtime?

Use Object Viewer — right-click any object in the Deployment view and select "View in Object Viewer." Then open a watch list (XML file) to see live attribute values with their quality status. Good quality data confirms the deployment is working correctly.

Q4: What is the purpose of the Check-in dialog after saving an object?

The Check-in dialog allows you to enter a descriptive comment about the changes made to the object. While optional, it is recommended for tracking version history and maintaining an audit trail of development operations.
