✍️ L34: Signed Writes

AVEVA™ Operations Management Interface 2023 — Module 8: Security

Pages 477–480 (Sections 8-45 through 8-48)

Introduction

Operators interact with automation objects through their individual attributes. These attributes can be assigned security classifications that determine the level of authentication required for operators to write to them at runtime. Security classifications provide a fine-grained mechanism to control who can modify process values and under what conditions.

Security Classifications for Object Attributes

A user-writable attribute listed in the Object Editor can be modified by operators at runtime and has an associated security control. This security control determines the runtime security classification. If the classification is configurable, you click the Security Control icon and make a selection from the available options.

Security Classification dropdown — The shield icon next to an attribute's Initial value field reveals the security classification options.
ClassificationRequirementDescription
FreeAccessNoneAnyone can write to the attribute
OperateCan Modify "Operate" AttributesRequires the Operate operational permission
SecuredWriteCan Modify "Operate" Attributes + authenticationRequires user name and password authentication for each write
VerifiedWriteCan Verify WritesRequires verification by a user with "Can Verify Writes" permission
TuneCan Modify "Tune" AttributesRequires the Tune operational permission
ConfigureCan Modify "Configure" AttributesRequires the Configure operational permission
ViewOnlyNone (read-only)Operators can only read the attribute at runtime
Key Concept: The seven security classifications provide a spectrum from no restrictions (FreeAccess) to read-only (ViewOnly), with SecuredWrite and VerifiedWrite providing the strongest protection by requiring explicit authentication for each write operation.

Secured Writes and Verified Writes

You can assign the SecuredWrite or VerifiedWrite security classification to a configurable attribute. These classifications require authentication to perform runtime writes. With authentication, users can write to such attributes through any assignment in a script (such as A=B) or any action in an animation graphic that alters the attribute value (such as a user input, slider, or other control).

Operators can write to SecuredWrite or VerifiedWrite attributes even if another user is logged on, without affecting that user's session. For SecuredWrite, the operator must have the "Can Modify Operate Attributes" permission. For VerifiedWrite, the operator must have "Can Modify Operate Attributes" and a separate verifier must have "Can Verify Writes."

Important: SecuredWrite and VerifiedWrite features only work when Galaxy security is enabled. They do not function if Galaxy security is set to None.

Secured Write Dialog

At runtime, when an operator attempts to write to a SecuredWrite attribute, a Secured Write dialog box appears. It shows the attribute name, the target value, and provides fields for entering operator credentials and comments.

Secured Write dialog — Shows the attribute name, target value, and fields for domain, username, password, and comments.

Verified Write Dialog

The Verified Write dialog shows the same information but includes additional fields for a verifier's credentials. Both the operator and verifier must be different users with the appropriate permissions.

Verified Write dialog — Includes both operator and verifier credential sections, plus a comments field for audit purposes.

Signed Write Events

Following a Secured Write or Verified Write operation, a security event is written to the event log. This event includes the event type stamp, operator name, verifier name (if any), the type of write (Secured or Verified), any comment entered by the user, and a reason description. These logged events can be viewed using the AlarmApp for audit and compliance purposes.

Key Concept: Signed writes create a complete audit trail. Every Secured Write and Verified Write is logged with the operator's identity, verifier's identity (if applicable), timestamp, and any comments — providing full traceability for critical process changes.

Knowledge Check

Q1: What is the difference between SecuredWrite and VerifiedWrite?

SecuredWrite requires the operator to re-enter their credentials (username and password) for each write operation. VerifiedWrite requires both an operator and a separate verifier — two different users must authenticate for the write to proceed.

Q2: What happens if Galaxy security is set to None when using SecuredWrite?

SecuredWrite and VerifiedWrite features do not work when Galaxy security is set to None. These security classifications require Galaxy security to be enabled.

Q3: Can an operator write to a SecuredWrite attribute while another user is logged in?

Yes. Operators can write to SecuredWrite or VerifiedWrite attributes even if another user is logged on. This does not affect the session of the logged-on user.

Q4: What information is logged after a Signed Write event?

The event log includes: event type stamp, operator name, verifier name (if any), type of write (Secured or Verified), comment entered by the user, and a reason description.

Q5: What is the ViewOnly classification?

ViewOnly makes an attribute read-only at runtime. Operators can read the value but can never write to it, regardless of their permissions.
