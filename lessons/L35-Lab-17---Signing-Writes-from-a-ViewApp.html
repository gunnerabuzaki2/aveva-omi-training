Lab Introduction
In this lab, you will configure security classifications for attributes in the $Mixer.Agitator template. You will set the Cmd attribute to SecuredWrite and the Speed.SP attribute to VerifiedWrite. During testing, you will attempt to write to attributes as users with different operational permissions, and observe the Secured Write and Verified Write dialog boxes in action.

Objectives

  • Configure security classifications for attributes in automation objects
  • Operate Secured Writes and Verified Writes as users with required permissions
  • Use the Secured Write and Verified Write dialog boxes in a running ViewApp

Attribute Configuration Summary

$Mixer.Agitator AttributeSecurity ClassificationMeaning
CmdSecuredWriteRequires operator authentication for each start/stop command
Speed.SPVerifiedWriteRequires both operator and verifier authentication for speed changes

Part 1: Configure Attributes with Signed Write Classifications

Step 1. On the Templates tab, double-click $Mixer.Agitator to open it in the Object Editor.
Step 2. On the Attributes tab, select the Cmd attribute.
Step 3. In the right pane, click the Security Classification icon (shield) next to the Initial value field, and select SecuredWrite.
Cmd attribute — Click the shield icon and select SecuredWrite from the dropdown. This requires authentication for every start/stop command.
Step 4. Click the Lock icon to propagate changes to derived objects.
Step 5. Select the Speed.SP attribute and set its Security Classification to VerifiedWrite.
Speed.SP attribute — Select VerifiedWrite. This requires both an operator and a different verifier to authenticate.
Step 6. Click the Lock icon to propagate changes. Save, close, and check in the template.

Part 2: Unlock Attributes for Runtime Modification

Steps 7–11. Reopen $Mixer.Agitator. Select the Cmd attribute and click the Lock icon to unlock it. This allows the attribute value to be changed at runtime.
Unlock the Cmd attribute — Click the padlock icon to allow runtime value changes. Locked attributes revert to their initial value at runtime.
Steps 12–14. Select Speed.SP and unlock it as well. Save, close, and check in the template.
Unlock the Speed.SP attribute — Both Cmd and Speed.SP must be unlocked for the signed write tests to work.

Part 3: Redeploy Objects

Step 15. In the Deployment view, right-click AppEngine1 and select Deploy with default settings.
Deployment complete — Eight Agitator objects are deployed with the updated security classifications.

Part 4: Test Secured Writes in Preview Session

Step 17. Switch to the ViewApp preview session. Navigate to Mixer100 and select Agitator_001.
Step 20. On the command panel, click Stop. The Secured Write dialog appears with the current user pre-filled.
Secured Write dialog — Appears when attempting to stop the agitator. The current logged-in user is pre-filled in the Domain and User name fields.
Step 21. Enter <domain>\karent as the user with a password, and add the comment "Command agitator to stop".
Enter verifier credentials — Use karent (Plant Supervisors 1) for the Secured Write. Note: karent does NOT have the Can Modify "Operate" Attributes permission needed for the Cmd attribute.
Step 22. Click Set. Notice the command panel shows a security error because karent does not have the proper permissions.
Security error — The red shield icon indicates the Secured Write failed because karent lacks the required "Can Modify Operate Attributes" permission for the Cmd attribute.
Step 23. Click Stop again. Enter <domain>\johnj (Plant Operators 1) with a password and comment.
Second attempt — Use johnj (Plant Operators 1) who has the required Can Modify "Operate" Attributes permission.
Step 25. Click Set. The agitator stops successfully with no security error.
Agitator stopped — The Secured Write succeeded because johnj has the correct permissions. Speed shows 0.00 (stopped).
Step 26. Click Start. In the Secured Write dialog, enter <domain>\lisay with comment "Command agitator to start".
Start agitator — Use lisay (Plant Operators 1) to authenticate the start command.
Agitator running — The start command succeeded. Level is at 31.67 and agitator speed is shown.

Part 5: Test Verified Writes

Step 29. In the command panel, enter a speed setpoint of 70 and press Enter. The Verified Write dialog appears.
Verified Write dialog — Requires both operator and verifier credentials. The verifier must be a different user than the operator.
Step 30. Enter <domain>\johnj for both the operator and verifier with passwords, and add the comment "Reset the setpoint as operators".
Same user for both roles — This will fail because operator and verifier must be different users.
Step 31. Click Set. The dialog displays an error: "Operator and Verifier have to be two different users".
Validation error — The system enforces two-person authentication by requiring different users for operator and verifier roles.
Step 32. Change the verifier to <domain>\lisay and click Set.
Correct configuration — johnj as operator and lisay as verifier. Both have appropriate permissions.
Key Takeaway: This lab demonstrated the two critical security features: SecuredWrite (single-user authentication) and VerifiedWrite (two-person authentication). The Cmd attribute uses SecuredWrite so any start/stop command requires re-authentication. The Speed.SP attribute uses VerifiedWrite so critical setpoint changes require two authorized individuals — preventing single-point-of-failure in process control.

Knowledge Check

Q1: Why did the first Secured Write attempt fail when using karent?

karent is in the Plant Supervisors 1 group, which has the "Can Verify Writes" permission but NOT the "Can Modify Operate Attributes" permission required to write to the Cmd attribute (which is an Operate-classified attribute).

Q2: Why can't the same user be both operator and verifier in a Verified Write?

The system enforces two-person authentication — requiring different users for operator and verifier prevents a single person from making critical changes alone, adding a layer of safety and accountability.

Q3: What was the correct user for the Secured Write stop command?

johnj (Plant Operators 1) — this group has the "Can Modify Operate Attributes" permission needed for the Cmd attribute's SecuredWrite classification.

Q4: What permissions are needed for a VerifiedWrite on Speed.SP?

The operator needs "Can Modify Operate Attributes" and the verifier needs "Can Verify Writes". Both must be different users with appropriate Galaxy security permissions.
